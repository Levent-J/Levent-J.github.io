总结目前常见的Crash日志收集工具

发表于 2018-12-13 | 分类于 工作总结
最近项目中需要加入一个Crash日志收集功能，要求接入简单且侵入性低，因此需要调研一下目前市面上国内国外的一些常见的日志收集工具，这里做一些总结。

Bugly

优点

  1. 接入简单
  2. 崩溃后立即上传，速度快
  3. 支持多种错误类型，支持ANR
  4. 自带运营统计
  5. 检索和查看很详细
  6. 支持自定义标签、渠道、用户
  7. 可以捕捉JS异常
  8. 支持日志

缺点

网易云捕

优点

  1. 接入简单
  2. 崩溃后立即上传，速度快
  3. 支持多种错误类型，支持ANR
  4. bug统计很详细
  5. 支持自定义标签、渠道、用户
  6. 体积很小
  7. 检索和查看很详细
  8. 支持日志、面包屑

缺点

友盟U-APP

优点

  1. 提供了随时开关的接口
  2. 有很好的运营统计

缺点

  1. 再次启动后才会上传
  2. 文档太简略
  3. 主要功能太简单

BugTags

优点

  1. 可由用户自己提交bug
  2. 可以上传截图
  3. 可配置项丰富

缺点

  1. 接入麻烦
  2. 上传慢
  3. 看起来更适合产品测试时使用，不适合上线后的统计

蒲公英

优点

  1. 接入简单
  2. 可由用户自己提交bug

缺点

  1. 文档比较乱，更像是一个用于团队项目管理的工具

Test Bugout

优点

  1. 支持多种错误类型，支持ANR 2.统计、问题检索都很详细
  2. 支持自定义标签、渠道、用户
  3. 支持很多测试相关功能
  4. 支持上传截图
  5. 支持日志
  6. 用户可自己提交bug

缺点

CrashEye

优点

  1. 接入简单
  2. 支持自定义标签、渠道、用户
  3. 可以捕捉脚本异常
  4. 监控功能很灵活
  5. 每一个设备都有专属的设备ID

缺点

  1. 接入只能需要手动导入jar包
  2. 只能区分JAVA和NDK

LogReoirter

优点

  1. 一个Github库
  2. 可以以邮件或Issue的形式上传崩溃日志
  3. 可配置项很多，后期可以按需修改框架自定义配置

缺点

Flurry

优点

  1. 统计、分析等功能很全面
  2. 界面设计炫酷
  3. 崩溃日志分析的图表很详细

缺点

  1. 崩溃情况反馈挺及时，但是崩溃日志上传极慢，且不稳定
  2. 缺少高级检索，错误种类也无法细分，只能做大致统计和简单的信息查看

Fabric Crashlytics

优点

  1. 接入简单
  2. 界面设计酷炫
  3. Crash统计图表比较详细
  4. 支持自定义键值对、用户、日志、主动上传异常
  5. 检索方式较灵活

缺点

  1. 只有再次启动后才会上传崩溃日志，且速度较慢

ACRA

优点

  1. Github开源库
  2. 支持多种上传日志方式

缺点

  1. 接入麻烦
  2. 只提供了日志的收集功能，需要自己提供接受服务器或邮箱等，因为也就缺少了日志的统计、检索，这些都需要我们自己去实现

CrashReporter

优点

  1. Github开源库
  2. 支持将日志以为文件形式保存在本地

缺点

  1. 缺少日志上传、统计分析功能，更适用于开发者测试
